threadly-2.3.0.jar

A library of java tools to assist with development of concurrent java applications. This is designed to be a complement to java.util.concurrent and uses java.util.concurrent to help assist in it's implementations where it makes sense.

These tools range from a unique priority based, and highly performant thread pool. To the ability to create a hierarchy of thready pools (sub pools in threadly named "Limiters"). To the ability to control threading and allow things to easily run single threaded where it makes sense. This library also includes some utilities to assist in debugging and bringing concurrent designs under test.

To get a more complete overview of our features take a look at our features wiki page.

It does include some new features, most prominently is the improved shutdown logic around SingleThreadScheduler, which now has the ability to "shutdownNow". * February 10th - Released version 1.0.0 of threadly. This release is very exciting to me as it indicates a new level for the project. A new level of stability and maturity, a new level of control and openness. As in my previous post you will note that we removed and changed a lot more than we originally expected to. I felt this was important because with the 1.0.0 release a lot of changes are occurring. A stricter versioning is part of this, allowing people to easily know what level of risk and change is included in every version (as described here on the wiki). We also plan to provide more visibility through pull requests, and filed issues into what exactly has changed (though the change log has always been very complete). We had lots of things in the 0.9 release we decided to remove for 1.0.0 because we felt that they added a lot of complexity, and code, when their value was minimal at best. Because it will be harder to remove those things in the future we decided now is the time to do it. So they have been removed, but can be easily added back if others find any of those features useful (they were removed because no one was using them and we saw minimal reason someone would start using them). Please look at the changelog on the wiki for a more complete list of what has changed.

* January 2nd - No new releases, but thought it was worth making a post about some changes occurring for the 1.0.0 release. We decided we want to thin things down, and reduce some complexity before our 1.0.0 release. Because of that we removed the following: VirtualLock, VirtualRunnable, VirtualCallable, TestablePriorityScheduler (which previously would take multi-threaded code and make it single threaded), FutureFuture, FutureListenableFuture, and combined SubmitterSchedulerLimiter and SimpleSchedulerLimiter into a single class "SchedulerLimiter". I realize this is a bit more change than we wanted to do before our 1.0.0 release but after lots of consideration I think now is the right time. This dramatically reduces the size and complexity. In addition AFAIK no one was currently using any of these features (probably because they are not very useful). The only exception that may have been the TestablePriorityScheduler which had the unique ability of taking multi-threaded code and making it single threaded. This was a hard decision because that seems like a valuable feature for unit testing. The reason we decided to remove it for now is because of the complexity it added. Further more it required you to use our locking scheme so that we can handle blocking correctly between threads. I don't particularly like the idea of having to change the run time code like this in order to support testability. If anyone would like to see these features back (or any other new features), please submit a request via the Issues on github. The great thing about git is that nothing is lost, and we can bring them back without too much headache if there is a demand. But for now, we figured it is better to reduce our API to what we know is useful and being used, and only add as needed. These changes have been pushed to sonatype under the version "1.0.0-SNAPSHOT".
